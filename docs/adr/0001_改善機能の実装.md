# ADR 0001: 改善機能の実装

## ステータス

提案中

## 背景

このシステムには、設計承認差し戻し、最終差し戻し、PR コメント分析など、ユーザが追加コメントを与えて AI を再実行する導線が既にある。
ただし、現状のコメントはジョブ単位の一時的な指示としてしか扱われず、次回以降の設計・実装・レビューには自動で引き継がれない。
また、ユーザ入力の具体的な指示を、将来も使える汎用的な方針へ言い換えて蓄積する仕組みがない。

このため、同じ種類の指摘を何度も繰り返す必要がある。
特に以下のような指示は、単発の差し戻しではなく、以後の実行で継続的に参照されるべきである。

- 画面レイアウトや文言配置の方針
- 設計書の粒度や追加で作成すべき補助資料
- ある実装変更に連動して必要になるテストデータや補助ファイルの更新方針
- リポジトリ固有のコーディング上の注意点やレビュー観点

一方で、単発ジョブの差し戻しコメントと、将来のジョブ全体に効かせる恒久的な改善指示は、寿命も用途も異なる。
両者を同じ保存先、同じ承認フロー、同じ読み取り方法で扱うと、意図しない再利用や追跡困難が起きる。

この機能の目的は、ジョブ単位のコメントをそのまま保存することではなく、AI によって抽象化し、再利用可能な改善方針へ変換することにある。
たとえば、XX画面のレイアウトを `ボタンを左に配置し、右に文言を配置` という入力に対して、その場限りの UI 修正指示ではなく、`操作要素を左、補足説明を右に配置する` のような画面レイアウト方針として保存する。

## 課題

- ジョブ単位の具体的なコメントを、継続利用できる汎用的な改善方針へ変換する仕組みがない
- 改善指示の版管理、差分確認、レビュー、履歴追跡の場所がない
- 実装用ブランチに改善指示を混在させると、コード変更と運用知識の変更が同じ差分に混ざる
- 各ワーカーがどの指示を、どのフェーズで、どの順序で読むかが定義されていない
- 複数ジョブから改善指示が更新された場合の競合方針がない

## 決定

### 1. 改善指示は恒久的なリポジトリ知識として管理する

本機能はオプション機能とし、デフォルトでは無効とする。
リポジトリごとに設定画面から有効化した場合のみ、改善用ワーカー、改善ブランチ、改善指示の読み込みを有効にする。

改善機能で承認された内容は、単発ジョブの再実行メモとして消費せず、リポジトリ単位で継続利用する。
ただし、ユーザコメントをそのまま恒久化するのではなく、AI が汎化して改善方針として整理し、ユーザ承認後にのみ保存する。
AI が恒久的な改善不要と判断した場合は、改善案を生成せずに終了できるものとする。

### 2. 保存先は実装ブランチとは分離した専用ブランチにする

改善指示は、対象リポジトリの既定ブランチや作業ブランチには直接保存せず、専用ブランチに保存する。

採用方針:

- ブランチはリポジトリごとに 1 本持つ
- ブランチ名は `improvement` を既定とする
- ブランチが存在しない場合は初回承認時に作成する
- 改善指示の更新はこのブランチへの commit / push として記録する

既定値は持つが固定ではない。
設定画面から、少なくとも以下をリポジトリ単位で変更可能にする。

- 改善機能の有効 / 無効
- 改善ブランチ名
- 改善指示ディレクトリ
- 承認前の作業用ディレクトリ

これにより、コード変更と改善知識を分離しつつ、Git の履歴、差分、復元、競合解消をそのまま利用できる。
この専用ブランチは repository worker の `作業ディレクトリ` に checkout し、VS Code などから直接編集できるようにする。
改善内容の編集や commit は、専用の改善用ワーカーが `作業ディレクトリ` を使って行う。
承認前の改善案本文も `作業ディレクトリ` に配置し、人間が VS Code のチャット機能や Codex などを使って直接修正できるようにする。

### 3. 改善指示は Markdown ファイルとして管理する

改善指示は人間が読めて、かつ AI がそのまま文脈として取り込める形式にする。
そのため、保存形式は Markdown を基本とし、必要なメタデータは front matter で持つ。

既定レイアウト:

- ブランチ: `improvement`
- ディレクトリ: `.improvements/`
- ファイル: `.improvements/*.md`

これらの既定値は設定画面から上書き可能とする。

各ファイルは以下の情報を持つ。

- `id`: 識別子
- `title`: 改善指示の要約
- `scope`: 適用範囲
- `phases`: `design` / `implementation` / `fix` / `review`
- `status`: `active` / `inactive`
- `updatedAt`: 最終更新時刻
- `source`: どのジョブ、どのコメント、どの承認から生成されたか
- 本文: AI に与える実際の指示

Markdown を採用する理由は以下。

- 人間がレビューしやすい
- AI 実行時にそのまま投入しやすい
- 変更差分が見やすい
- 将来、ルール説明や補足例を文章で増やしやすい

`.improvements` は改善指示の正本であり、VS Code などで人間が直接編集する対象でもある。
ただし、`.improvements/*.md` 自体をすべての AI provider が自動検出して読む前提にはしない。
承認前の入力情報や下書きは `.improvements` には保存しない。
承認前の下書き本文は `作業ディレクトリ` に置き、承認済みの正本だけを `.improvements` に反映する。

### 4. ジョブ単位の差し戻しコメントとは分離する

ジョブ単位の差し戻しコメントや PR コメント分析結果は、改善方針の元になる入力であり、そのジョブの再実行にのみ強く効く一時的な指示としても扱う。
これらは従来どおりジョブ成果物やイベント履歴に残す。

一方、改善機能で承認された恒久指示は、元コメントを AI が汎化した結果として別保存で管理し、以後のジョブで参照する。

優先順位は以下とする。

1. 現在のジョブに対する明示的なユーザコメント
2. 現在のジョブで承認された設計・修正案
3. 改善ブランチ上の恒久的な改善指示
4. スキル定義や通常の issue / PR 文脈

上位の指示が下位と矛盾する場合は、上位を優先する。

### 5. 各ワーカーは実行前に改善指示を読み込む

repository worker は通常のソース checkout とは別に、改善ブランチの内容を取得し、AI 実行時の文脈に含める。

AI への受け渡し方法は provider ごとに異なるため、`AGENTS.md` 固定にはしない。
改善指示の正本は `.improvements/*.md` とし、実行時は provider adapter がその内容を各 provider に適した形へ変換して渡す。

例:

- `AGENTS.md` を読む provider には、作業ディレクトリ上に `AGENTS.md` を生成または追記する
- `CLAUDE.md` のような provider 固有ファイルを使う provider には、その形式へ変換する
- ファイル自動読込がない provider には、prompt 本文または system prompt に改善指示を注入する

つまり、改善機能が保証するのは `.improvements` の保存形式までであり、AI への最終的な受け渡しは provider adapter の責務とする。

改善機能が無効なリポジトリでは、provider adapter は改善指示の読込や実行時注入を行わない。

適用方針:

- `design` ワーカーは `design` 向け改善指示を読む
- `implementation` ワーカーは `implementation` 向け改善指示を読む
- `pr_feedback` の修正実装は `fix` 向け改善指示を読む
- `review` ワーカーは `review` 向け改善指示を読む

フェーズに加えて、必要なら以下の条件で絞り込む。

- repository 単位
- job type 単位
- パスや機能領域などのタグ

初期実装では、repository 単位 + phase 単位の適用に留める。
条件分岐が増えすぎると、何が効いているか追跡しにくくなるためである。

provider adapter がファイル経由で指示を渡す場合は、少なくとも以下を含める。

- この実行で有効な改善指示の一覧
- phase ごとの適用対象
- 指示の優先順位
- 元ファイルへの参照

ジョブ固有の差し戻しコメントや PR コメント分析結果は、恒久指示とは混ぜず、従来どおり prompt 本文側で渡す。
provider 固有ファイルには恒久指示だけを書く。

### 6. 改善案の生成は改善用ワーカーが担当する

改善案を生成するのは、設計ワーカーや実装ワーカーではなく、専用の改善用ワーカーとする。

理由:

- 設計・実装・レビューの本来責務と、改善方針の抽象化責務を分離するため
- 改善案生成時には、コード変更よりもコメントの汎化、既存方針との重複確認、更新対象の選定が重要になるため
- 改善ブランチの編集、commit、競合解消は通常のジョブワーカーとは別の制御が必要になるため

各ワーカーの役割は以下とする。

- `design` ワーカー: issue や既存成果物をもとに設計成果物を生成する
- `implementation` ワーカー: 承認済み設計や既存修正内容をもとに実装を行う
- `review` ワーカー: PR 文脈をもとにレビュー成果物を生成する
- `improvement` ワーカー: ユーザコメントや PR コメント分析結果を汎化し、必要な場合のみ恒久的な改善方針案を生成し、承認後に改善ブランチを更新する

改善用ワーカーは `作業ディレクトリ` 上の改善ブランチを使って動作する。
設計・実装ワーカーは改善指示を読むだけで、改善ブランチ自体の更新は行わない。
改善機能が無効な場合、`improvement` ワーカーは起動しない。

### 7. 改善機能の承認対象は「改善案」そのものとする

ユーザ入力のコメントをそのまま保存するのではなく、AI が汎化・構造化した改善案を承認対象にする。

理由:

- ノイズの多い口語コメントをそのまま永続化しないため
- 具体的な局所指示を、再利用可能な方針へ言い換えるため
- 重複した指示を整理できるため
- 将来の AI 実行時に、そのまま読み込める品質の文章へ変換するため

たとえば、`ボタンを左に配置し、右に文言を配置` という入力を、そのまま保存するのではなく、`画面上の操作要素は左、説明文は右に配置する` といった方針へ変換して保存する。

承認後の保存対象は、AI が生成した改善案 Markdown とする。
恒久化すべき改善がないと判断した場合は、`改善不要` を結果として記録し、承認対象を作らない。

### 8. 改善指示の更新は上書きではなく履歴付きで行う

同じ論点の改善指示が再度提案された場合、単純に別ファイルを増やすのではなく、既存ファイルを更新する。
ただし Git 履歴で差分は残す。

更新単位:

- 同一 `id` の改善指示は更新扱い
- 類似だが別論点の指示は新規追加

初期実装では、既存指示との対応付けは AI 任せにしすぎず、以下のどちらかに限定する。

- UI で既存指示を明示選択して更新する
- タイトルや `id` をキーに機械的に更新対象を決める

意味的類似度だけで自動マージする方式は採用しない。
誤更新時の影響が大きいためである。

## 詳細設計

### 保存構造

改善ブランチのファイル構成は以下を基本とする。

```text
.improvements/
  README.md
  ui-layout-policy.md
  testdata-update-policy.md
  review-checklist.md
```

各ファイルの例:

```md
---
id: ui-layout-policy
title: 画面のボタンと説明文の左右配置方針
scope: repository
phases:
  - design
  - implementation
status: active
updatedAt: 2026-06-07T00:00:00Z
source:
  jobId: issue-...
  event: improvement_approved
---

- ボタンを左、補足説明を右に配置する。
- 一覧やカードでは、操作要素より説明文を後に置く。
- モーダルでは、開いた時点で主要情報が展開済みで見えることを優先する。
```

承認前の編集用ファイルは、改善ブランチを checkout した `作業ディレクトリ` に保存する。

```text
source/<repo>-improvement/.improvement/
  input.md
  context.json
  draft.md
  notes.md
```

役割は以下とする。

- `input.md`: ユーザ入力コメント、PR コメント、関連する明示指示
- `context.json`: 汎化対象のメタデータ、関連 job、対象 phase、既存改善指示の参照情報
- `draft.md`: 承認前の改善方針案。本ファイルを VS Code や Codex で直接編集する
- `notes.md`: 人間や AI が追記する補足メモ

典型的なユースケースは、`improvement` ワーカーが `draft.md` を生成した後、ユーザが VS Code のチャット機能や Codex を使って `draft.md` を修正し、その内容を承認対象にする流れである。

承認前後の監査用スナップショットとログは、ジョブ成果物ディレクトリに保存する。

```text
artifacts/<repo>/jobs/issue_<number>/improvement/
  input.md
  draft.md
  approval.json
  decision.json
  result.md
  stdout.log
  stderr.log
```

役割は以下とする。

- `input.md`: 承認対象を生成した時点の入力スナップショット
- `draft.md`: 承認対象として提示した時点の draft スナップショット
- `approval.json`: 承認、却下、コメント差し戻しの状態
- `decision.json`: `draft_created` / `no_improvement_needed` / `approved` / `rejected` などの最終判定
- `result.md`: 承認済みの最終改善案、または却下時の最終提案内容
- `stdout.log` / `stderr.log`: 改善用ワーカーの実行ログ

`improvements` ブランチへ反映するのは、`approval.json` 上で承認された `result.md` を `.improvements/*.md` に整形した内容だけとする。
`decision.json` が `no_improvement_needed` の場合は、`improvements` ブランチへは何も反映しない。

### 読み取り方法

各ワーカーは実行開始時に以下を行う。

1. 作業対象リポジトリの通常 checkout を準備する
2. 改善ブランチを fetch する
3. `.improvements/*.md` を読み取る
4. 現在の phase に該当する `active` な指示だけを抽出する
5. provider adapter が抽出結果を provider ごとの入力形式へ変換する
6. ファイル指示型の provider では、ワーカー作業ディレクトリのソースルートに必要な指示ファイルを生成または更新する
7. prompt 注入型の provider では、改善指示を prompt 本文または system prompt に追加する
8. ジョブ固有コメントは別途 prompt 本文に追加する

改善指示はソースコードとして checkout されたブランチへ merge しない。
また、provider adapter が生成した実行用ファイルを既定ブランチへ commit しない。
AI 実行の入力文脈としてのみ利用する。

既存の provider 固有指示ファイルが対象リポジトリに存在する場合は、以下の順序で扱う。

1. リポジトリに元からある指示ファイル
2. `.improvements` から生成した管理下セクション
3. ジョブ固有コメントの prompt

つまり、改善機能は既存の provider 固有指示ファイルを破壊せず、管理下のセクションだけを追加または更新する。
ファイル自動読込を持たない provider では、同じ内容を prompt に重ねて渡す。

### 作成フロー

改善機能の最小フローは以下とする。

1. 設定画面で対象リポジトリの改善機能を有効にする
2. ユーザがコメントを入力する
3. 入力内容を `作業ディレクトリ/.improvement/input.md` へ保存し、監査用にジョブ成果物ディレクトリにもスナップショットを残す
4. `improvement` ワーカーが入力コメント、既存改善指示、関連成果物を読み取る
5. 恒久化すべき改善がある場合は `作業ディレクトリ/.improvement/draft.md` を生成し、無い場合は `decision.json` に `no_improvement_needed` を記録して終了する
6. `draft.md` が生成された場合のみ、ユーザは VS Code、VS Code のチャット機能、Codex などを使って `draft.md` を必要に応じて修正する
7. 承認時点の `draft.md` をジョブ成果物ディレクトリへスナップショットし、承認または却下の結果を `improvement/approval.json` に保存する
8. 承認された場合のみ、`improvement` ワーカーが `improvement/result.md` をもとに改善ブランチ上の Markdown を追加または更新する
9. 次回以降の該当 phase 実行時に各ワーカーが自動で読み込む

改善機能が無効なリポジトリでは、このフロー自体を開始しない。

### 設定項目

設定画面から変更可能な項目は以下とする。

| 項目 | 既定値 | 説明 |
| --- | --- | --- |
| 改善機能 | `disabled` | リポジトリ単位で改善機能を有効 / 無効にする |
| 改善ブランチ名 | `improvement` | 承認済み改善指示を保持するブランチ名 |
| 改善指示ディレクトリ | `.improvements/` | 承認済み改善指示の保存先 |
| 承認前作業ディレクトリ | `.improvement/` | `draft.md` など承認前ファイルの配置先 |

設定値はリポジトリ単位で保持し、既定値を使いたい場合は空欄で既定値へフォールバックしてもよい。

### 画面構成方針

本機能の UI は、`設定画面` と `改善点画面` の 2 系統に分ける。

設定画面の方針:

- 改善機能は repository ごとの設定項目として扱う
- `有効 / 無効` を最上段に置き、無効時は関連設定を非活性または折りたたみ表示にする
- `改善ブランチ名`、`改善指示ディレクトリ`、`承認前作業ディレクトリ` は同じセクションにまとめる
- 既定値がある項目はプレースホルダまたは補助文で既定値を示す
- `作業ディレクトリ` と `承認前作業ディレクトリ` の役割の違いが分かる説明を付ける

改善点画面の方針:

- 改善点は専用画面で一覧表示する
- 一覧には、対象リポジトリ、元コメントの要約、draft のタイトル、状態、最終更新時刻を表示する
- 一覧の 1 行をクリックすると、ポップアップで詳細を表示する
- ポップアップには、元コメント、現在の `draft.md`、承認結果、関連 job を表示する
- `draft.md` はポップアップ内でブラウザ編集できるようにする
- 承認ボタンはポップアップ画面内に置く
- 却下、再生成、保存も同じポップアップ画面内から行えるようにする
- VS Code での編集も継続して許可するが、ブラウザ内編集を主導線として持つ
- `draft.md` の保存先が `作業ディレクトリ` であることは詳細画面に明示する
- `no_improvement_needed` と判定された項目も一覧に残し、改善不要で終了したことを確認できるようにする
- `no_improvement_needed` の項目では、ポップアップに改善不要の理由を表示し、承認ボタンは表示しない

表示の優先順位は以下とする。

1. ユーザが入力した元コメント
2. `improvement` ワーカーが生成し、必要に応じて人間が修正した `draft.md`
3. 承認結果
4. 承認後に `.improvements/*.md` へ反映される最終結果

この構成により、設定画面は機能有効化とパス設定、改善点画面は一覧・詳細・承認、`作業ディレクトリ` は編集対象、ジョブ成果物ディレクトリは監査記録、という責務分離を明確にする。

### ワーカー構成

改善機能を含む worker 構成は以下とする。

| ワーカー | 主な入力 | 主な出力 | 書き込み先 |
| --- | --- | --- | --- |
| `design` | issue、本ジョブのコメント、既存の改善指示 | 設計成果物 | ジョブ成果物ディレクトリ |
| `implementation` | 承認済み設計、本ジョブのコメント、既存の改善指示 | 実装成果物、パッチ、テスト結果 | ジョブ成果物ディレクトリ |
| `review` | PR 文脈、既存の改善指示 | レビュー成果物 | ジョブ成果物ディレクトリ |
| `improvement` | ユーザコメント、PR コメント分析結果、既存の改善指示 | 汎化した改善方針案、中間成果物、改善指示の更新 commit | `作業ディレクトリ/.improvement/`、ジョブ成果物ディレクトリの `improvement/`、`作業ディレクトリ` 上の改善ブランチ |

設計・実装・レビューの各ワーカーは「改善指示の利用者」であり、改善用ワーカーは「改善指示の生成者兼更新者」である。

### 競合時の扱い

改善ブランチ更新時に push 競合した場合は、以下で処理する。

1. 最新の改善ブランチを fetch する
2. 対象ファイルを再読込する
3. 同一ファイルに衝突がなければ再 commit / push する
4. 同一ファイルに衝突がある場合は自動解決せず失敗扱いにする
5. ユーザに再確認または手動調整を促す

## 実績

現在の実装で入っている内容と、ADR 本文との差分は以下。

### 実装済み機能

- リポジトリ単位の改善設定を実装済み
  - `improvementEnabled`
  - `improvementBranch`
  - `improvementDir`
  - `improvementWorkDir`
- 改善点画面を実装済み
  - 一覧表示
  - 詳細ポップアップ
  - ブラウザ内での draft 編集
  - 承認
  - `relatedJobId`、`decisionReason`、`phases` の表示
- ジョブ詳細画面から改善案生成を開始可能
- 改善案生成後、承認時に改善ブランチへ commit / push する実装を追加済み
- 承認済み改善方針は front matter 付き Markdown で保存する
- `no_improvement_needed` を実装済み

### 現在のフォルダ構成

実装上の改善関連ファイルは、現在は次の構成になっている。

```text
source/<repo>-improvement/
  .improvement/
    input.md
    issue_<issue番号>_<タイトル>.md
  .improvements/
    issue_<issue番号>_<タイトル>.md

artifacts/<repo>/jobs/issue_<issue番号>/improvement/
  input.md
  context.json
  draft.md
  result.md
  approval.json
  decision.json
  git-fetch.log
  git-checkout.log
  git-reset.log
  git-branch.log
  git-write.log
  git-add.log
  git-commit.log
  git-push.log
```

役割は以下。

- `work/.improvement/`
  - 承認前の作業用領域
  - 現在の実装では `draft.md` 固定ではなく、`issue_<issue番号>_<タイトル>.md` で保存している
- `work/.improvements/`
  - 承認済み改善方針の正本
  - front matter 付き Markdown を保存する
- `jobs/.../improvement/`
  - 監査用スナップショットと承認履歴
  - 生成時の入力、draft、承認結果、改善ブランチ反映ログを保持する

### ADR 記載との差分

- ADR では承認前作業ファイルを `work/.improvement/draft.md` としていたが、実装は `work/.improvement/issue_<issue番号>_<タイトル>.md`
- ADR では `notes.md` を想定していたが、現時点では未実装
- ADR では `stdout.log` / `stderr.log` を改善成果物に記載しているが、現実装では改善承認の Git 操作ログを `git-*.log` として保存している
- ADR では改善用ワーカーが改善ブランチを更新すると記載しているが、現実装では Web の承認処理から app 層を呼び出して更新している
- ADR では provider adapter による改善指示読込を前提にしているが、現時点では保存・承認・一覧表示までが実装済みで、各 provider 実行時の自動読込までは未実装

### 保存形式の実績

承認済み改善方針は、現在は次のような front matter を持つ。

```md
---
id: issue-109
title: ロジック分割方針
scope: repository
phases:
  - implementation
  - fix
status: active
updatedAt: 2026-05-20T09:58:30Z
source:
  jobId: fixture-pr-comment-analysis-ready
  issueNumber: 109
  repository: coco-papiyon/korobokcle
  event: improvement_approved
---
```

`phases` は固定値ではなく、改善本文と元コメントの文脈から推定して保存する実装になっている。

### テストデータの実績

`create-testdata` では改善機能も含めて再現できる。

- `fixture-pr-created`
  - `draft_created`
- `fixture-pr-comment-analysis-ready`
  - `approved`
- `fixture-failed`
  - `no_improvement_needed`

これにより、改善点画面の 3 パターンを `tests/data` だけで確認できる。

改善指示はコードよりも運用知識に近く、誤った自動マージのコストが高い。
そのため、保守的に扱う。

### 監査性

改善指示は少なくとも以下を追跡できる必要がある。

- どのジョブから発生したか
- どのコメントや承認操作に基づくか
- いつ追加または更新されたか
- どの phase に効くか

この追跡情報は、承認前については `作業ディレクトリ/.improvement/` の編集内容とジョブ成果物ディレクトリの `improvement/` スナップショット、承認後については `.improvements/*.md` の front matter と Git 履歴の両方で残す。

## 代替案

### 1. 既定ブランチにそのまま保存する

不採用。

- コード変更と運用知識の変更が同じ履歴に混ざる
- 改善指示だけを確認、巻き戻し、比較しづらい
- 実装 PR に無関係な差分が混ざりやすい

### 2. 実装用ブランチごとに改善指示を持つ

不採用。

- ジョブ間で再利用されない
- 恒久知識として蓄積できない
- 同じ改善指示が複数ブランチへ分散する

### 3. アプリ内部 DB にだけ保存する

不採用。

- リポジトリと一緒に履歴管理しづらい
- レビューしづらい
- 将来、外部ツールや手作業で編集しにくい
- AI が参照する知識がリポジトリ外へ分散する

### 4. YAML だけで管理する

今回は不採用。

- 機械処理はしやすいが、説明文が長くなると読みにくい
- AI に渡す本文が結局文字列の塊になる

メタデータ量が増えて Markdown では扱いにくくなった段階で再検討する。

## 影響

### 利点

- 同じ指摘を繰り返さずに済む
- リポジトリ固有の知見を蓄積できる
- 何が AI の挙動に影響しているか Git 上で追跡できる
- コード変更と改善知識を分離できる

### 欠点

- 改善指示の管理対象が増える
- 指示が増えすぎると AI 文脈が肥大化する
- 良くない改善指示を恒久化すると、後続ジョブに継続して悪影響が出る

そのため、指示件数の上限、無効化、見直し導線は別途必要になる。

## 今回のスコープ外

- 複数 repository にまたがる共有改善指示
- 改善指示の自動分類や自動統合
- 改善指示にスクリプトや実行可能コードを含めること
- 改善指示の優先度計算やスコアリング
- 改善指示専用の新しいジョブ種別追加

初期実装では、既存の設計承認系フローを流用して改善案を承認し、保存後に各ワーカーが読む構成を優先する。

## 実装時の注意

- Phase 6 では shared な `作業ディレクトリ` を改善ブランチ専用 checkout とし、`worker-*/source` は通常ソース実行専用 clone とする方針で分離した。改善同期時は `.improvement/` を `git clean` の除外対象にして承認前 draft を保護する
- provider ごとの入力形式、生成ファイル名、既存ファイルとのマージ方法、管理セクションの書式を別途決める必要がある
- 既存の rerun comment と恒久改善指示を UI 上で混同しないよう、表示と命名を分ける必要がある
- PR コメント分析結果をそのまま改善指示に昇格させる場合は、ユーザが「今回だけ」と「今後も適用」を選べる必要がある

## 結論

改善機能は、ジョブ単位の差し戻しコメントを補完するための、リポジトリ単位の恒久知識管理機能として実装する。
保存先は対象リポジトリの専用改善ブランチとし、内容は `.improvements/*.md` で管理する。
改善内容は `作業ディレクトリ` 上で人間も改善用ワーカーも編集できるようにする。
AI 実行時は、その内容を provider adapter が provider ごとの形式へ変換して読ませる。

これにより、AI 実行の改善を継続的に蓄積しつつ、コード変更・ジョブ成果物・恒久指示を明確に分離できる。
