# ループエンジニアリング支援の不足機能整理

- Status: Accepted (実装ジョブの基本ループを実装済み)
- Date: 2026-07-04
- Scope: `korobokcle` 全体

## Context

`korobokcle` は、Issue / PR を監視して設計・実装・レビューを進める土台を持っている。
また、`worktree`、Markdown 成果物、承認フロー、スキル生成の仕組みもすでに存在する。

一方で、現在の実装は「1回の AI 実行を回して成果物を出す」ことに寄っており、反復改善のためのループ制御は弱い。
そのため、ループエンジニアリングを支援する道具として見ると、次のような機能が不足している。

## ループエンジニアリングで実装すべき内容

ループエンジニアリングでは、単発の成果物生成ではなく、実装と検証の往復を前提にした制御と記録が必要になる。

最低限、次の機能を揃える。

1. 反復回数の管理
2. 実装者エージェントと検証者エージェントの役割分離
3. 検証結果に応じた再実装分岐
4. 試行ごとの履歴とログの保存
5. 失敗理由の分類
6. 人間の承認や介入
7. 試行状況の可視化
8. 再現性のための設定・プロンプト固定

## 現在の実装状況

次の表は、ループ支援で必要な機能を、`korobokcle` の対応状況とあわせて整理したもの。

| 必要な機能 | 概要 | いまの対応状況 | 補足 |
| --- | --- | --- | --- |
| 反復回数の管理 | 何回まで実装・検証を繰り返すかを制御する | 対応済み | `implementationLoopCount` を設定として持ち、設定画面の「実装・検証ループ回数」から変更できる。既定値と上限もある |
| 実装者 / 検証者の役割分離 | 実装者が変更し、検証者が確認する構成にする | 対応済み | 実装者と検証者を別プロセス・別ログで扱う基盤がある |
| 検証結果に応じた再実装分岐 | `passed` / `changes_requested` に応じて次の行動を変える | 対応済み | 検証結果を受けて再実装または終了へ進める |
| 試行ごとの履歴保存 | 各回の入力、出力、差分、テスト結果を追えるようにする | 部分対応 | 役割別ログと成果物はあるが、専用の試行履歴モデルはない |
| 失敗理由の分類 | 失敗原因を機械的に扱えるカテゴリへ分ける | 部分対応 | ログには残るが、分類コードとしての標準化は限定的 |
| 人間の承認 / 介入 | 途中停止や再試行の判断を人間が行えるようにする | 部分対応 | 承認アクションはあるが、ループ途中の介入操作は弱い |
| 試行状況の可視化 | 何回目の試行か、どこで失敗したかを見える化する | 部分対応 | ジョブ詳細でログと成果物は見られるが、試行タイムラインはない |
| 再現性のための設定・プロンプト固定 | 実行条件を固定し、同じ試行を再現できるようにする | 部分対応 | スキル生成や設定はあるが、試行単位で完全に固定する仕組みはない |

## Decision

`korobokcle` には、AI 実行を単発処理ではなく「試行の連続」として扱うためのループ基盤を追加する。
ただし ADR では具体的な内部実装方式までは固定しない。
ここで決めるのは、実装で満たすべき機能と、各段階で確認すべきテスト項目である。

追加対象は、次の 5 つを優先する。

1. 反復制御
2. 試行履歴の保持
3. 評価基準の構造化
4. 失敗分類と次アクションの分岐
5. 可視化と手動介入

## Missing Features

### 1. 試行履歴の保持

必要なもの:

- 各試行の入力、出力、差分、テスト結果、エラーを保存する
- 試行ごとの `prompt`、`model`、`branch`、`worktree` を追跡する
- 前回の失敗と今回の修正点を比較できるようにする

不足理由:

- 役割別ログと成果物はあるが、試行単位で永続的に追跡する専用モデルはまだない

### 2. 評価基準の構造化

必要なもの:

- 成功条件を機械可読で持つ
- テスト、静的解析、レビュー指摘、手動確認を別々に評価する
- 合格条件をループ継続条件と連動させる

不足理由:

- 検証者の応答やテスト項目はあるが、評価基準を共通ルールとして保存・再利用する形にはなっていない

### 3. 失敗分類と次アクション

必要なもの:

- コンパイル失敗
- テスト失敗
- 仕様誤解
- 差分不足
- ファイル配置ミス
- 権限や環境要因

それぞれに対して、再プロンプト、修正、停止、手動介入のどれを行うかを変える。

不足理由:

- 現状はログに失敗内容が残るだけで、次のアクションに使う分類コードとしては整理しきれていない

### 4. 可視化と手動介入

必要なもの:

- 試行の一覧表示
- 成功 / 失敗理由の時系列表示
- 途中停止、再試行、別分岐の開始
- 人間が判断した理由の記録

不足理由:

- ジョブ詳細ではログを確認できるが、試行タイムラインや介入履歴をまとめて追える UI はない

### 5. 再現性のための設定・プロンプト固定

必要なもの:

- スキルの版
- プロンプトの版
- 評価ルールの版
- 実行環境の版

不足理由:

- 生成時点の設定はあるが、試行ごとの差分を追跡できるほど固定化されていない

### 6. 人間の承認を含む例外処理

必要なもの:

- 途中停止
- 再試行
- 手動差し戻し
- 成功 / 失敗の最終確定

不足理由:

- 承認フローは存在するが、ループ途中での例外判断を継続的に扱う導線はまだ薄い

### 補足

- `design_running`、`implementation_running`、`review_running` のような実行中状態でも、ジョブ詳細のログ表示は可能。
- ジョブ詳細 API は `workspace/<repo-id>/<job-id>/logs` を読み、役割別ログを返す。
- ただし、これは「見える化」であり、「試行履歴の永続モデル」そのものではない。

## Design

### 1. 反復実行を前提にした実装フローを追加する

実装フェーズは、1 回の生成で終わらず、検証結果に応じて再実装できる流れにする。

必要な機能:

- 実装と検証を反復できる
- 途中で人間が介入できる
- 同じ失敗を繰り返した場合に停止できる
- 反復回数に上限を持てる

注記:

- 反復回数の管理、実装者 / 検証者の分離、再実装分岐は既に実装済みの前提とする。
- ここで追加で詰めるのは、履歴、評価、失敗分類、可視化の不足分である。

### 2. 実装で保持すべき情報を定義する

実装では、各試行について最低限次の情報を保持する。

- 入力
- 出力
- 使用したモデル
- 実行したテスト項目
- 判定結果
- 次に行うべきアクション

この情報があれば、実装者エージェントと検証者エージェントを分けても、あとから試行の流れを追える。

### 3. テスト項目を設計要件として明示する

各試行の後に、次の順で評価する。

1. 構文・ビルド
2. テスト
3. 差分妥当性
4. 人間の確認

評価結果は「合格 / 再試行 / 差し戻し / 停止」を返せるようにする。

設計書には、具体的な実装コードではなく、確認すべきテスト項目を列挙する。

- どのコマンドを実行するか
- どの状態で合格とみなすか
- どの失敗を再実装対象にするか
- どの失敗を人間確認に送るか

注記:

- 実装者 / 検証者の分離自体は既にあるため、この節では検証基準の明文化に焦点を置く。

### 4. エージェント分離は実装方針として扱う

実装時には役割を分けることを推奨する。

- 実装者エージェント: 変更を加える役割
- 検証者エージェント: テストと判定を行う役割

ただし、ADR では具体的なプロンプト構成や内部クラス構成までは決めない。
それらは実装設計で調整可能にする。

### 5. 失敗分類を標準化する

失敗は文字列ログではなく、分類コードとして保持する。

- `compile_error`
- `test_failure`
- `semantic_mismatch`
- `missing_file`
- `permission_denied`
- `environment_issue`
- `unknown`

分類ごとに、次の試行で使う補助情報を変える。

- エラーメッセージの要約
- 該当ファイル
- 前回との差分
- 再実行するコマンド

### 6. UI とログを試行中心にする

画面には、ジョブの現在状態だけでなく、試行のタイムラインを表示する。

- 何回目の試行か
- どこで失敗したか
- 何を直したか
- 次に何をするか

また、画面上では次を分けて表示する。

- 実装者エージェントの出力
- 検証者エージェントの判定
- 人間の承認状態

### 7. 再現性のための設定・プロンプト固定

改善ループの再現性を上げるため、試行時点の入力条件を固定できるようにする。

- スキルの版
- プロンプトの版
- 評価ルールの版
- 実行環境の版

注記:

- これは今の設定管理の延長ではなく、試行ごとの比較を可能にするための追加要素として扱う。

## Considered Options

### Option A: 現状の単発実行を維持する

利点:

- 実装が簡単
- 今の設計をほぼ変えない

欠点:

- 改善の再現性が低い
- 失敗の学習が進まない
- ループエンジニアリングの目的に届かない

### Option B: ジョブごとに個別の再実行ロジックを増やす

利点:

- 既存フローに足しやすい

欠点:

- 設計が分岐しやすい
- 実装の重複が増える
- 失敗分類や履歴が統一されない

### Option C: 共通ループ基盤を導入する

利点:

- 設計・実装・レビュー・コンフリクト解消で同じ反復モデルを使える
- 履歴、評価、停止条件を統一できる
- 将来の改善がしやすい

欠点:

- 初期実装のコストが高い
- 既存の状態管理との整合が必要

採用判断:

- Option C を採用する

## Consequences

### Positive

- 反復改善の経過を追えるようになる
- 失敗を学習資産にしやすくなる
- 人間が途中介入しやすくなる
- どの改善が効いたかを比較できる

### Negative

- DB と UI のモデルが増える
- ログと成果物の管理が複雑になる
- 既存ジョブ状態との整合調整が必要になる

## Priority

優先順位は次の通り。既に対応済みの反復回数管理や役割分離ではなく、まだ部分対応にとどまる領域を優先する。

1. 試行履歴
2. 評価基準
3. 失敗分類
4. 可視化と手動介入
5. 版管理と比較分析

## Next Steps

- `job` と `attempt` のデータモデルを定義する
- 試行履歴を保存する永続化層を設計する
- 評価結果と失敗分類を共通のルールとして整理する
- UI に試行タイムラインと介入履歴を追加する
- スキル、プロンプト、実行環境の固定方法を検討する

## Implementation Note

最初の実装対象として、Issue の実装ジョブに次の基本ループを導入する。これは既に対応済みの実装フローを前提に、残る可視化や履歴の不足を補うための土台でもある。

1. ジョブworkerの開始時に、実装者用と検証者用の Agent CLI を別プロセスとして起動する
2. 実装者エージェントが専用 worktree を直接変更する
3. 検証者エージェントが同じ worktree を読み取り、設計・リポジトリ指示に基づくテストと判定を行う
4. 検証結果が `changes_requested` の場合、その指摘を次の実装者プロンプトへ渡す
5. 検証結果が `passed` の場合は早期終了し、ジョブを承認待ちへ進める
6. 設定した最大回数まで合格しない場合は、検証指摘をエラーとして残してジョブを失敗にする

反復回数は設定 `implementationLoopCount` で指定する。既定値は3、設定可能な上限は10とする。

実装者と検証者は同じプロバイダー・モデル設定を利用するが、セッションと役割プロンプトは分離する。検証者の応答は連携を安定させるため、`passed` または `changes_requested` を含むJSON形式とする。

この段階では試行履歴の専用データモデルとUIタイムラインは導入せず、役割別プロセスログとIssue単位のAIログへ記録する。これらは上記 Next Steps で拡張する。
