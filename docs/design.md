# korobokcle 再設計

## 1. 目的

`korobokcle` は、GitHub Issue / Pull Request を監視し、設計・実装・レビューを AI と人間の承認で進めるローカルツールである。
フロントエンドは Vue + TypeScript で実装する。

今回の再設計では、次の方針を採る。

- 設計と実装を別ジョブとして扱う
- issue / PR の状態タグを見て、AI 実行の入口を決める
- 設計やレビューの確認結果は人間が画面で承認する
- 実装時だけ `work_dir` 配下に worktree を作る
- 状態の表示は日本語、内部管理は英語にする
- 1 プロセスは 1 リポジトリのみを対象にする

## 2. 前提

### 2.1 ディレクトリの役割

- `base_dir`
  - ツールを実行したときのカレントディレクトリ
  - 引数で指定可能
  - 修正対象のソースコードが置かれているローカルリポジトリ
  - 設計・レビューの参照元であり、設計結果の保存先でもある
- `tool_dir`
  - `korobokcle` 本体の配置ディレクトリ
  - 実行バイナリである `korobokcle` が格納されているディレクトリを `tool_dir` とする。
  - 実行バイナリ、静的コンテンツ、プロンプトを持つ
- `work_dir`
  - `korobokcle` が使用するデータを格納するディレクトリ  
  - デフォルトでは、 `tool_dir` = `work_dir`
  - 引数で指定可能
  - 設定、DB、作業用ディレクトリ、ログを持つ
  - 実装時の worktree はここに作る

### 2.2 重要な制約

- 1 プロセスで扱うリポジトリは 1 つだけ
- リポジトリ設定も 1 つだけ保持する
- AI に「待つ」「監視する」役割を持たせない
- 設計や実装のスキルはリポジトリに格納されたものを使う
- プロンプトの指示は最小限にする
- 状態の内部表現は英語の enum、画面表示は日本語にする

### 2.3 用語

| 用語 | 意味 |
| --- | --- |
| ジョブ | 1 回の AI 実行単位。設計、実装、レビュー、レビュー指摘対応を別々のジョブとして扱う |
| ワーカー | 1 ジョブを処理する goroutine |
| 状態タグ | GitHub Issue / PR に付与する状態表現のラベル |
| 作業用 worktree | 実装時に `work_dir` 配下へ作る git worktree |
| 成果物 | `base_dir/.workspace/...` に保存する Markdown や補助ファイル |

## 3. 全体構成

```text
main
  ├─ config / db 初期化
  ├─ GitHub poller
  ├─ job dispatcher
  ├─ job worker manager
  ├─ approval API / Web UI
  └─ artifact / log / skill loader
```

責務分離は次の通り。

- 監視
  - Issue / PR の状態タグを読む
  - 対象をジョブ化する
- 実行
  - AI を起動する
  - 成果物を作る
  - worktree を更新する
- 承認
  - 画面で結果を表示する
  - OK / NG を受け付ける
  - GitHub へ投稿やラベル更新を行う
- 永続化
  - ジョブ状態
  - 承認履歴
  - 実行ログ
  - 成果物パス

## 4. ディレクトリ構造

### 4.1 `base_dir`

`base_dir` は対象リポジトリのルートであり、設計とレビューの文脈を保持する場所でもある。

```text
base_dir/
  .workspace/
    design/
    implementation/
    review_fix_design/
    review_fix_implementation/
    review/
    pr_conflict/
```

保存ファイルは次の方針とする。

| 用途 | 保存先 |
| --- | --- |
| 設計結果 | `base_dir/.workspace/design/<issue番号>_<issueタイトル>.md` |
| 実装結果 | `base_dir/.workspace/implementation/<issue番号>_<issueタイトル>.md` |
| レビュー指摘の設計結果 | `base_dir/.workspace/review_fix_design/<issue番号>_<issueタイトル>.md` |
| レビュー指摘の実装結果 | `base_dir/.workspace/review_fix_implementation/<issue番号>_<issueタイトル>.md` |
| PR レビュー結果 | `base_dir/.workspace/review/<issue番号>_<issueタイトル>.md` |
| PR コンフリクト解消結果 | `base_dir/.workspace/pr_conflict/<PR番号>_<PRタイトル>.md` |

注記:

- ファイル名のタイトルは GitHub の issue / PR タイトルを安全に正規化する
- `issue番号` が取れない場合は PR 番号を代替値にする

### 4.2 `tool_dir`

`tool_dir` はアプリ本体の実行基盤（実行バイナリ、静的コンテンツ、プロンプト等）である。
実行バイナリが格納されているディレクトリを `tool_dir` とする。

```text
tool_dir/
  prompt/
  static/
```

役割は次の通り。

- `prompt/`
  - AIへ渡す編集可能なプロンプトテンプレート
  - `*.tmpl` はスキル生成のたびに読み込むため、変更時の再ビルドは不要
  - Goの `text/template` 形式で、既存のテンプレート変数は維持する
- `static/`
  - Web UI の静的ファイル

### 4.3 `work_dir`

`work_dir` はアプリが使用するデータの格納場所であり、実装時の worktree はここに作る。
デフォルトでは、`work_dir` = `tool_dir`であり、起動引数で変更可能とする。

```text
work_dir/
  config/
  db/
  workspace/
  logs/
```

役割は次の通り。

- `config/`
  - アプリ設定
  - リポジトリ設定
  - 監視設定
    - ジョブ多重度（全ジョブ共通、デフォルト4）
- `db/`
  - SQLite
- `workspace/`
  - 実装時の worktree
- `logs/`
  - ワーカー単位のログ

### 4.4 実装時 worktree

実装ジョブだけは、`work_dir/workspace/` 配下に worktree を作成する。

```text
work_dir/workspace/<repo-id>/<job-id>/worktree/
```

方針:

- 1 ジョブ = 1 worktree
- worktree は job ごとに分離する
- ブランチ名は job 生成時に確定し、job 本体に保存する
- 画面表示や再取得時は保存済みの branch をそのまま使う
- 設計やレビューでは worktree を作らない

## 5. 状態モデル

### 5.1 表示と内部管理

- 画面表示
  - 日本語
- 内部管理
  - 英語の enum

例:

| 表示 | 内部 |
| --- | --- |
| 設計中 | `design_running` |
| 設計完了 | `design_ready` |
| 設計承認済み | `design_approved` |
| 実装中 | `implementation_running` |
| 実装完了 | `implementation_ready` |
| 実装承認済み | `implementation_approved` |
| PR済み | `pr_created` |
| レビュー指摘あり | `pr_review_comment` |
| コンフリクト検知済み | `pr_conflict` |
| コンフリクト解消中 | `pr_conflict_running` |
| コンフリクト解消完了 | `pr_conflict_ready` |
| コンフリクト解消済み | `pr_conflict_resolved` |
| レビュー指摘検討済み | `review_fix_design_ready` |
| レビュー検討承認済み | `review_fix_design_approved` |
| レビュー指摘修正済み | `review_fixed` |
| レビュー中 | `review_running` |
| レビュー完了 | `review_ready` |
| レビュー承認済み | `review_approved` |

### 5.2 GitHub 上の状態タグ

GitHub の Issue / PR には、内部状態に対応するラベルを付与する。
ラベル名は英語で統一し、画面だけ日本語で見せる。

ラベルは 1 対 1 で固定する。

## 6. ジョブ種別

### 6.1 Issue 系

Issue は次のジョブに分割する。

1. 設計ジョブ
2. 実装ジョブ
3. レビュー指摘対応の設計ジョブ
4. レビュー指摘対応の実装ジョブ

各ジョブは、それぞれ専用のワーカー goroutine を持つ。

### 6.2 PR 系

PR は次のジョブに分ける。

1. PR レビュージョブ
2. レビュー却下後の再実行は、Issue 側のレビュー指摘対応ジョブとして扱う
3. PR コンフリクト解消ジョブ

## 7. 監視ルール

### 7.1 Issue

#### 7.1.1 指定のタグがない

処理:

1. 実装検討として設計ジョブを起動する
2. 設計結果を `base_dir/.workspace/design/<issue番号>_<issueタイトル>.md` に保存する
3. 状態タグを設計済みにする
4. 画面に設計内容を表示する
5. OK なら承認、NG なら今の結果と指示内容を渡して再設計する
6. 設計を承認したら Issue に設計内容を投稿する
7. 状態を設計承認済みにする

#### 7.1.2 状態 = 設計承認済み

処理:

1. 設計に従い実装ジョブを起動する
2. `work_dir/workspace/` 配下に worktree を作る
3. 実装結果を `base_dir/.workspace/implementation/<issue番号>_<issueタイトル>.md` に保存する
4. 状態タグを実装済みにする
5. 画面に実装内容を表示する
6. OK なら承認、NG なら今の結果と指示内容を渡して再実装する
7. 実装を承認したら状態を実装承認済みにする

#### 7.1.3 状態 = 実装承認済み

処理:

1. 設計内容を含めて PR を作成する
2. PR 本文の末尾に `Closes #<issue番号>` を付けて、マージ時に issue を自動クローズできるようにする
3. 状態を PR 済みにする

#### 7.1.4 状態 = PR レビューコメント状態

処理:

1. PR から実装内容と指摘を取得する
2. レビュー指摘対応の設計ジョブを起動する
3. 設計結果を `base_dir/.workspace/review_fix_design/<issue番号>_<issueタイトル>.md` に保存する
4. 状態タグをレビュー指摘検討済みにする
5. 画面に設計内容を表示する
6. OK なら承認、NG なら今の結果と指示内容を渡して再設計する
7. 設計を承認したら Issue に設計内容を投稿し、状態をレビュー検討承認済みにする

#### 7.1.5 状態 = レビュー検討承認済み

処理:

1. 設計に従い実装ジョブを起動する
2. 実装結果を `base_dir/.workspace/review_fix_implementation/<issue番号>_<issueタイトル>.md` に保存する
3. 状態タグを実装済みにする
4. 画面に実装内容を表示する
5. OK なら承認、NG なら今の結果と指示内容を渡して再実装する
6. 実装を承認したら状態を実装承認済みにする

#### 7.1.6 状態 = 実装承認済み

処理:

1. PR に修正内容をコメント投稿する
2. 状態をレビュー指摘修正済みにする

### 7.2 PR

#### 7.2.1 指定のタグがない

処理:

1. PR のレビューを行う
2. レビュー結果を `base_dir/.workspace/review/<issue番号>_<issueタイトル>.md` に保存する
3. 状態タグをレビュー済みにする
4. 画面にレビュー内容を表示する
5. OK なら承認、NG ならレビューコメントを PR に送信し、紐づく Issue の状態を PR レビューコメント状態にする
6. レビューを承認したら状態をレビュー承認済みにする

#### 7.2.2 状態 = レビュー指摘修正済み

処理:

- 指定のタグがない場合と同じ

#### 7.2.3 PR がコンフリクトしている

処理:

1. `mergeable=CONFLICTING` または `mergeStateStatus=DIRTY` のPRを検知する
2. PRのhead/baseブランチ名、競合ファイル、関連PR情報を取得する
3. ブランチ名にIssue番号が含まれる場合は、双方のIssue本文・コメント・ラベルを取得する
4. headブランチのworktreeを作成し、baseブランチをmergeして競合を展開する
5. コンフリクト解消スキルを使い、双方のIssueの意図を確認して競合を修正する
6. 結果を `base_dir/.workspace/pr_conflict/<PR番号>_<PRタイトル>.md` に保存し、画面で承認または再実行できるようにする
7. 承認時に修正をcommitしてheadブランチへpushし、PRへ結果をコメントして状態をコンフリクト解消済みにする

## 8. ワーカー設計

### 8.1 基本方針

ワーカーは 1 ジョブ 1 goroutine とする。

- 1 ジョブにつき 1 ワーカーを起動する
- ジョブが終わったらワーカーは停止する
- そのジョブが完了したら次のジョブは別ワーカーで起動する

### 8.2 最大数

同時に起動できるワーカー数は設定可能にする。

- 設計系ワーカー上限
- 実装系ワーカー上限
- PR レビュー系ワーカー上限

### 8.3 生成ルール

ワーカー管理は次の流れとする。

1. poller が issue / PR を取得する
2. 状態タグを見て起動対象を判定する
3. まだ空きがある場合だけ goroutine を起動する
4. ワーカーは DB 上の job を占有する
5. 終了時に占有を解除して停止する

### 8.4 停止条件

ワーカーは次で停止する。

- 成果物保存が完了した
- 承認待ち状態へ遷移した
- 失敗状態になった
- キャンセルされた

## 9. AI 実行方針

### 9.1 スキル利用

設計、実装、レビュー、PRコンフリクト解消は、リポジトリに格納されたスキルを使う。スキル生成画面ではコンフリクト解消用スキルも生成・再生成対象として選択できる。

本アプリが渡す指示は最小限とし、次だけを渡す。

- 対象フェーズ
- Issue / PR のメタデータ
- 既存成果物
- 承認コメント
- worktree パス
- 出力先パス

### 9.2 フェーズ別入力

#### 設計

- Issue 本文
- ラベル
- コメント
- 関連 PR 情報
- 既存設計があればそれ

#### 実装

- 設計成果物
- Issue 情報
- 既存コード
- 実装指示コメント
- worktree

#### PR レビュー

- PR 差分
- Issue 設計
- 実装成果物
- テスト結果

#### レビュー指摘対応の設計

- PR コメント
- レビュー結果
- Issue 設計
- PR 差分

#### レビュー指摘対応の実装

- レビュー指摘対応の設計
- PR コメント
- worktree

### 9.3 出力ルール

AI の主出力は Markdown とする。

- 設計は設計書
- 実装は実装要約
- レビューは指摘一覧と判断根拠
- すべて保存先を明示する

## 10. 承認フロー

### 10.1 画面表示

各成果物は画面に表示し、次の 2 択を取る。

- OK
- NG

NG の場合は、ユーザの指示文を追加で受け取り、再実行する。

### 10.2 GitHub 反映

承認が通ったら、次を行う。

- Issue への設計投稿
- Issue 状態タグ更新
- PR へのレビューコメント投稿
- 必要に応じた PR 作成 / 更新

### 10.3 再実行

再実行時は、前回の成果物と人間の追加指示をそのまま次回入力に含める。
これにより、差分だけを直すのではなく、意図を失わずに再設計・再実装できる。

## 11. 永続化

### 11.1 SQLite

SQLite には次を保存する。

- ジョブ
  - branch
- 状態遷移
- 監視結果
- 承認結果
- ワーカーの起動 / 終了記録
- GitHub への投稿結果

### 11.2 Markdown 成果物

Markdown は人間がレビューしやすい正本として扱う。

- `base_dir/.workspace/...` に置く
- UI だけでなく、エディタでも読める
- 再実行時の入力にも再利用する

### 11.3 ログ

ログは `work_dir/logs/` に保存する。

- ワーカー単位で分ける
- job ID を含める
- 実行中の AI 入出力と Git 操作の追跡に使う

## 12. まとめ

この再設計の中心は次の 4 点である。

1. 設計と実装を分ける
2. issue / PR の状態タグで処理を進める
3. 実装時だけ `work_dir` 配下に worktree を切る
4. AI はリポジトリ内スキルを最小指示で実行する

これにより、設計結果のレビュー、実装結果のレビュー、PR コメント起点の再設計を独立に扱える。
