## 処理内容

- ADR(Architectural Decision Record)を読み取る。
  格納ディレクトリ: `docs/adr/<issue番号>_*.md` ※issue番号はゼロパディングありの数字
  ADRが存在しない場合は、ADRを読み取らず、issueの内容だけで設計を行う。
- issueおよびADRの内容をもとに、要件として整理する。
- issueの概要を簡潔にまとめる。必要に応じて、要件を満たすための制約条件や前提条件も明記する。
- 要件を満たすための設計案を作成する。
- 設計案に基づき、変更対象ファイルを整理する。
  設計書(`docs`)についても修正要否を判断し、修正対象及び、修正内容を明記する。
- 設計案に基づき、テスト計画を整理する。
  実装するテスト項目及び、テスト方法を決定する。
  Go言語を修正した場合は、 `go test ./...` を実行する
  npm(Vue.js)を修正した場合は `cd frontend && npm ci && npm test` を実行する。
- 設計案に基づき、リスクを整理する。
  リスクには、設計案の不備や、テスト計画の不備、実装上の不具合などが含まれる。
- 設計後、セルフレビューを行い、設計内容に不備がないか確認し、必要に応じて修正する。
- **必須出力形式** に従って、設計内容をまとめる。

## 必須出力形式
次の見出しで結果を出力する。

```
## 概要
## 要件
## 設計
## 変更対象ファイル
## テスト計画
## リスク
```
